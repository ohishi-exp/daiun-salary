# web地球号バグ: 1039(山口勝也) 2月22日 運転170分が消失

## 概要

ドライバー1039の2026年2月において、2/22 05:40〜08:30の運転時間170分がどの日にも計上されていない。

## 影響

- 運転合計: 170分(2:50)過少
- 拘束合計・実働合計・時間外合計にも同額の影響

## 詳細

### 運行データ

運行NO: `2602210830310000002838`（2/21 08:30 → 2/26 10:09）

```
2/21 08:30:31  運行開始
2/21 08:30〜23:17  運転・荷役・休憩（通常勤務）
2/21 23:17:20  休息302開始（383分 = 6時間23分）
2/22 05:40:41  休息終了 → 運転201開始（195分）
2/22 08:30:31  ← 始業から24時間経過
2/22 08:55:51  運転201終了
2/22 08:55〜20:20  運転・荷役・休憩（通常勤務）
2/22 20:20:02  休息302開始（1004分）→ 次の日へ
```

### 休息基準の判定

2/21 23:17の休息(383分)は以下のいずれも満たさない:
- [原則] 連続540分以上 → 383分 < 540分 ✗
- [例外] 連続480分以上 → 383分 < 480分 ✗
- [特例] 分割休息 → 1回目のみで累計不足 ✗

→ **休息基準未達のため、次の拘束開始を新規始業としない**

### web地球号の出力（参照CSV）

| 日付 | 始業 | 終業 | 運転 | 備考 |
|------|------|------|------|------|
| 2/21 | 08:30 | **23:17** | 13:40 (820分) | 前op 75分 + 現op 745分 |
| 2/22 | **08:30** | 20:20 | 5:26 (326分) | 08:30以降の運転のみ |

### 問題点

1. **2/21の終業が23:17**（休息開始時刻）になっている
   - 383分の休息は基準未達なので、終業扱いすべきでない
   - 24hルールでは始業(08:30)+24h=翌08:30が日締めのはず

2. **2/22の始業が08:30**（24h境界）になっている
   - 休息は05:40に終了しているが、始業は08:30

3. **05:40→08:30の170分が消失**
   - 2/21: 終業=23:17なので23:17以降は計上されない
   - 2/22: 始業=08:30なので05:40〜08:30は計上されない
   - → 170分の運転がどちらにも帰属しない

### イベント詳細（消失区間）

```
2/22 05:40:41〜08:55:51  運転(201) 195分
  うち 05:40:41〜08:30:31 = 約170分 → 計上されない
  うち 08:30:31〜08:55:51 = 約25分 → 2/22に計上（326-301=25）
```

## 期待される正しい動作

24hルールに従えば:
- 2/21のworkday: 08:30→翌08:30（24時間）
- 05:40→08:30の運転170分は**2/21のworkday内**
- 2/21の運転 = 820 + 170 = **990分(16:30)**

または休息開始で終業とするなら:
- 2/21のworkday: 08:30→23:17
- 2/22のworkday: 05:40→20:20（休息終了が新始業）
- 2/22の運転 = 326 + 170 = **496分(8:16)**

いずれにしても170分はどちらかの日に計上されるべき。

## デジタコ会社からの回答 (2026-03-18)

デジタコ会社に確認した結果、以下の回答を得た:

- **ルール**: 「始業から24時間後が休息中なら休息の開始が終業になる」
- **バグ確認**: 「休息終了が始業になっていないのはバグ」

つまり、web地球号では休息の開始(23:17)を終業とする動作は正しいが、
次の始業を休息の終了(05:40)ではなく24h境界(08:30)にしているのはバグ。

→ GitHub Issue #1 として追跡中

## 現状の対応

当システムでは24hルールに従い2/21に計上している（drive=496が2/22に帰属）。
web地球号の出力と2:50(170分)の差分が発生する。web地球号側のバグと確認済み。

CSV比較時にはこの差分を「既知バグ」として自動マークし、
累計・合計への連鎖差分も含めて known_bug として追跡する。
